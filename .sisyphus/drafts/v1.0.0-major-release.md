## Research Findings Summary

### 1. OpenTelemetry (Current State)

**What exists:**
- OTel SDK with OTLP exporter (Jaeger/Zipkin compatible)
- gRPC interceptors for automatic server/client spans
- CLI flags: `--tracing-enable`, `--tracing-endpoint`, `--tracing-sample-rate`, etc.
- Coordinator `Compile()` has comprehensive span
- Worker `Compile()` has comprehensive span

**Critical gaps:**
- `handleFlutterBuild()` — NO span
- `handleUnityBuild()` — NO span  
- `Worker.Build()` — NO span
- CLI commands (hgbuild) — NO application spans
- Individual executors — NO child spans (all black boxes)
- `--tracing-headers` flag missing

**Recommended improvements:**
1. Add spans to Flutter/Unity handlers
2. Add span to `Worker.Build()`
3. Add CLI command spans for client-side observability
4. Add executor child spans for granularity
5. Expose `--tracing-headers` CLI flag

### 2. TLS (Current State)

**What exists:**
- `internal/security/tls/` — cert loading, mTLS support
- CLI flags: `--tls-cert`, `--tls-key`, `--tls-ca`, `--tls-require-client-cert`
- TLS 1.2 minimum by default

**Critical gaps:**
- ❌ NO certificate rotation support (certs loaded once at startup)
- ❌ NO cert expiration validation
- ❌ NO path existence check before loading
- ❌ NO CRL/OCSP revocation support
- ❌ NO cipher suite configuration

**Recommended improvements:**
1. Certificate rotation via `GetCertificate` callback + file watcher
2. Certificate expiration validation on load
3. Path validation with better error messages
4. Add `--tls-min-version` CLI flag
5. SIGHUP handler to trigger cert reload

### 3. Lifecycle Cleanup (Current State)

**What exists:**
- All executors use `defer os.RemoveAll(workDir)` for task temp dirs
- Flutter executor has `defer stopGradleDaemon()`
- Docker executor has `defer ContainerRemove()`
- Cache uses `.tmp` suffix during writes

**Critical gaps:**
- ❌ NO cleanup if process crashes before defer runs
- ❌ NO periodic cleanup of orphaned temp directories
- ❌ NO cache `.tmp` file cleanup on startup
- ❌ NO tracking of cleanup failures
- ❌ Gradle daemons may orphan (best-effort cleanup)

**Recommended improvements:**
1. `LifecycleManager` with periodic `cleanupStaleTempDirs()`
2. Startup cleanup — sweep WorkDir for orphaned `hg-*-*` directories
3. Cache `.tmp` cleanup on startup and periodic sweeps
4. Track and log cleanup failures
5. Consider `--no-daemon` for Gradle to avoid orphaning

### 4. WAN Support (Current State)

**What exists:**
- mDNS discovery for LAN (`_hybridgrid-coord._tcp.local.`)
- `--coordinator` flag for explicit coordinator address
- `--advertise-address` for NAT override
- `InMemoryRegistry` with TTL heartbeat
- `DiscoverySource` field already supports "mdns", "wan", "manual"

**Critical gaps:**
- ❌ mDNS doesn't work across subnets/WAN
- ❌ No central registry for multiple coordinators
- ❌ No NAT traversal (STUN/TURN/ICE)
- ❌ No persistence — coordinator restart loses all workers
- ❌ No region/zone awareness

**Recommended approach (for low latency + high performance):**
1. **Central coordinator registry** — Workers connect directly to coordinator IP
   - Simple, works through NAT (worker initiates connection)
   - Low latency (direct gRPC, no relay)
   - Reuse existing gRPC connection for callbacks (reverse tunnel pattern)
2. Hybrid mode: mDNS for LAN + explicit `--coordinator` for WAN
3. Optional: Persistent registry (SQLite) for coordinator restarts

### 5. Rust/Go Distributed Builds

**Rust findings:**
- **sccache** is the standard (Mozilla, 7.1k stars, production-ready)
- Distributed mode: Scheduler + Build Servers + JWT auth
- Challenge: Incremental compilation ties cache to working directory
- Solution: `--remap-path-prefix` for path normalization

**Go findings:**
- Go is already fast; distributed compilation has diminishing returns
- Better approach: **Shared build cache** (gobuildcache pattern)
- Cache `GOCACHE` directory on coordinator, workers pull dependencies

**Recommendation:**
- **Rust**: Integrate with existing sccache or implement lightweight remote cache
- **Go**: Focus on cache sharing rather than distributed compilation
- Both: Executor that archives project, sends to worker, returns artifact

## Open Questions

1. **WAN persistence**: SQLite or in-memory with config file?
   - Recommendation: SQLite for simplicity (already Go stdlib via `modernc.org/sqlite`)

2. **Rust executor pattern**: Full project archive or sccache integration?
   - Recommendation: Full project archive (like Flutter/Unity) — simpler, works everywhere

3. **Cache eviction**: Time-based, size-based, or both?
   - Recommendation: Size-based LRU with configurable max size (already have `max_size_gb` in config)

## Technical Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| WAN discovery | Central coordinator | Low latency, simple, NAT-friendly |
| Rust builds | Full project archive | Consistent with Flutter/Unity pattern |
| Go builds | Full project archive | Cache dependencies on worker |
| Cleanup frequency | Periodic (5min default) | Balance between freshness and overhead |
| TLS rotation | File watcher + SIGHUP | Industry standard pattern |
| OTel scope | Add missing spans only | Don't break existing instrumentation |

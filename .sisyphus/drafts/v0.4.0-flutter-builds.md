## Research Findings

### Explore Agent: Existing Build Type Patterns
**Key Discovery: Flutter is ALREADY partially implemented in proto/capabilities!**

- Proto already defines `BUILD_TYPE_FLUTTER = 2` in `proto/hybridgrid/v1/build.proto`
- `FlutterConfig` message exists (target platform, build mode, flavor)
- `FlutterCapability` message exists (sdk_version, platforms, android_sdk, xcode_available)
- **Capability detection already implemented** in `internal/capability/detect.go:342-396`
  - Detects Flutter SDK via `flutter --version --machine`
  - Checks ANDROID_HOME for Android SDK
  - Checks xcodebuild on macOS for iOS/macOS
- **Registry matching already implemented** in `internal/coordinator/registry/registry.go:276-279`
  - `matchesCapability()` checks `caps.Flutter != nil`

**What's MISSING:**
1. Flutter executor (`internal/worker/executor/flutter.go`) — not implemented
2. `Build()` RPC handler — returns Unimplemented
3. CLI `hgbuild flutter` command — not implemented
4. Cache key generation for Flutter builds

### Librarian Agent: Flutter Build Process Analysis

**Distributable Stages:**
| Stage | Parallelizable | Notes |
|-------|----------------|-------|
| Dart Kernel Compilation | ✅ YES | Each `.dart` → `.dill` independently |
| Android NDK Build (per ABI) | ✅ YES | arm64-v8a, armeabi-v7a, x86_64 can run on separate workers |
| Asset Bundling | ✅ YES | No dependencies between assets |
| iOS IL2CPP → LLVM | ✅ YES | Each translation unit independent |
| CocoaPods Build | ✅ YES | Each pod independent |
| Gradle Build (per flavor) | ✅ YES | prod/dev/staging can parallelize |

**Sequential (NOT distributable):**
- Dart AOT final linking
- APK/AAB final assembly
- iOS Archive/Signing

**Platform Constraints:**
- iOS builds REQUIRE macOS (Apple requirement, no workaround)
- Android can build on Linux/macOS/Windows
- Code signing requires certificates on the build machine

## Worker Setup Recommendation (Prometheus)

Given CI/CD pipeline use case + all platforms, I recommend:

**Approach: Docker containers + Selective workers**

1. **Docker images for reproducibility** — Flutter SDK versions locked, consistent environments
2. **Selective workers** — iOS builds REQUIRE macOS workers (no Docker workaround)
3. **Capability-based routing** — Workers advertise which platforms they can build

Rationale:
- CI/CD needs reproducibility → Docker
- iOS code-signing requires macOS hardware → selective workers
- All-platforms means mixed worker types → capability matching

## Scope Boundaries
- **INCLUDE (v0.4.0)**:
  - Android builds only (APK/AAB)
  - All build modes (debug/profile/release)
  - Build flavors support
  - Whole-build distribution with extensible design for future stage splitting
  - Worker capability detection for Flutter/Android SDK
  - Cache integration for Flutter builds
  - CLI `hgbuild flutter` command
  
- **EXCLUDE (defer to v0.5.0)**:
  - iOS builds (requires macOS workers, code signing complexity)
  - Desktop builds (Linux/macOS/Windows)
  - Web builds
  - Fine-grained task splitting (Dart AOT, NDK per-ABI separately)

## Technical Decisions (Final)
- **Build granularity**: Whole-build distribution with extensible design
  - Rationale: Simpler to implement, still provides value for CI/CD parallelization (multiple apps/flavors in parallel)
  - Design interfaces for future fine-grained splitting
- **Worker setup**: Docker containers + selective workers
  - Docker images with Flutter SDK for reproducibility
  - Capability-based routing to Flutter-capable workers
- **Test strategy**: TDD (tests first)
  - Rationale: Go project already has test infrastructure, TDD prevents regressions

### Metis Review Decisions
- **Build artifacts**: Same cache system (content-addressable xxhash, consistent with C++)
- **Gradle daemons**: Fresh daemon per build (isolated, no state pollution)
- **Flutter SDK versions**: Docker images tagged with Flutter version (flutter:3.19, etc.)
- **Android signing**: Unsigned builds only (signing handled externally in CI/Fastlane)
- **Failure granularity**: Structured error types + full log (useful for debugging)

## Guardrails (from Metis Review)
- NO iOS support in v0.4.0 — defer all iOS-related code paths
- NO signing/keystore handling — builds are unsigned
- NO fine-grained task splitting yet — whole `flutter build` only
- NO FVM complexity — use Docker images for version control
- MUST use fresh Gradle daemon per build — no daemon reuse
- MUST preserve existing C++ functionality — no breaking changes

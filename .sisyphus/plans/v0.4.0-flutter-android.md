# v0.4.0: Flutter Android Build Support

## TL;DR

> **Quick Summary**: Add distributed Flutter Android builds to hybridgrid, enabling CI/CD pipelines to parallelize multiple app builds across workers using Docker containers with pinned Flutter SDK versions.
> 
> **Deliverables**:
> - Flutter executor for Android builds (APK/AAB)
> - `hgbuild flutter` CLI command
> - `Build()` RPC implementation in coordinator
> - Docker images with Flutter SDK
> - Cache integration for Flutter build outputs
> - Dashboard updates for Flutter build stats
> 
> **Estimated Effort**: Medium (7-10 tasks)
> **Parallel Execution**: YES - 3 waves
> **Critical Path**: Proto verification â†’ Executor â†’ RPC â†’ CLI â†’ E2E

## Context

### Original Request
Add Flutter as a new build type to the distributed build system for v0.4.0, focusing on Android builds with iOS deferred to v0.5.0.

### Interview Summary
**Key Discussions**:
- Target platforms: Android only (iOS deferred to v0.5.0 due to macOS-only requirement)
- Primary use case: CI/CD pipeline parallelization
- Build granularity: Whole-build distribution with extensible design

**Research Findings**:
- Flutter proto definitions ALREADY EXIST (BUILD_TYPE_FLUTTER, FlutterConfig, FlutterCapability)
- Capability detection ALREADY IMPLEMENTED in `internal/capability/detect.go`
- Registry matching ALREADY IMPLEMENTED in `internal/coordinator/registry/registry.go`
- MISSING: Executor, Build RPC, CLI command, cache integration

### Metis Review
**Identified Gaps** (addressed):
- Build artifact storage: Using existing xxhash cache (consistent with C++)
- Gradle daemon handling: Fresh daemon per build (isolation)
- Flutter SDK versioning: Docker images tagged with Flutter version
- Android signing: Unsigned builds only (external signing)

## Work Objectives

### Core Objective
Enable distributed Flutter Android builds (APK/AAB) across hybridgrid workers, with Docker-based Flutter SDK management and whole-build task distribution.

### Concrete Deliverables
- `internal/worker/executor/flutter.go` â€” Flutter executor implementation
- `internal/coordinator/server/grpc.go` â€” Build() RPC handler
- `cmd/hgbuild/main.go` â€” `hgbuild flutter` CLI command
- `internal/cache/key.go` â€” Flutter cache key generation
- `build/docker/flutter/Dockerfile` â€” Flutter Docker images
- `internal/observability/dashboard/api.go` â€” Flutter build stats

### Definition of Done
- [ ] `hgbuild flutter build apk --release` distributes to workers and returns APK
- [ ] Workers with Flutter capability receive Flutter tasks
- [ ] Workers without Flutter reject Flutter tasks gracefully
- [ ] Cache hits skip compilation and return cached APK
- [ ] Dashboard shows Flutter build statistics
- [ ] All tests pass: `go test -race ./...`

### Must Have
- Flutter executor that runs `flutter build apk/appbundle`
- Build() RPC routing for BUILD_TYPE_FLUTTER
- CLI command with --target (apk/appbundle), --build-mode (debug/profile/release), --flavor
- Docker image with Flutter SDK for worker containers
- Cache key generation for Flutter builds

### Must NOT Have (Guardrails)
- NO iOS builds â€” defer to v0.5.0
- NO desktop/web builds â€” Android only
- NO APK signing â€” unsigned builds only (CI handles signing)
- NO fine-grained task splitting â€” whole `flutter build` command only
- NO FVM integration â€” Docker images for version control
- NO Gradle daemon reuse â€” fresh daemon per build
- NO breaking changes to C++ compilation â€” preserve existing functionality

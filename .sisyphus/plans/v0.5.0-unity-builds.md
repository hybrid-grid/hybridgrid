### Unity Build Specifics
**How Unity batch mode works**:
```bash
# Headless build command
Unity -batchmode -quit -nographics \
  -projectPath /path/to/project \
  -executeMethod BuildScript.PerformBuild \
  -buildTarget Android \
  -logFile build.log
```

**Key considerations**:
- Unity Editor must be installed on worker (not containerizable without license complexity)
- License activation required (Pro/Plus license for CI, or manual seat per worker)
- Build targets: Android, iOS (macOS only), Windows, Linux, macOS, WebGL
- Multiple Unity versions may coexist (Unity Hub manages this)
- Build methods: custom C# `BuildScript.cs` in project, or standard BuildPipeline
- Scripting backends: Mono, IL2CPP (IL2CPP requires platform SDKs)

## Work Objectives

### Core Objective
Enable distributed Unity builds across hybridgrid workers with Unity Editor installed, supporting Android/iOS/Windows/Linux/macOS/WebGL targets via batch mode.

### Concrete Deliverables
- `internal/capability/detect.go` — `detectUnity()` function
- `internal/worker/executor/unity.go` — Unity executor implementation
- `internal/coordinator/server/grpc.go` — Build() RPC routing for Unity
- `cmd/hgbuild/main.go` — `hgbuild unity` CLI command
- `internal/cli/unity/` — Unity CLI package

- `internal/observability/dashboard/api.go` — Unity build stats
- `docs/unity.md` — Unity build documentation

### Definition of Done
- [ ] `hgbuild unity build --target android --project /path` distributes to workers and returns APK
- [ ] Workers with Unity capability receive Unity tasks
- [ ] Workers without Unity reject Unity tasks gracefully
- [ ] Cache hits skip compilation and return cached artifact
- [ ] Dashboard shows Unity build statistics
- [ ] All tests pass: `go test -race ./...`

### Must Have
- Unity executor that runs `Unity -batchmode` with proper flags
- Capability detection finding Unity Editor installations
- Build() RPC routing for BUILD_TYPE_UNITY
- CLI command with --target (android/ios/windows/linux/macos/webgl), --build-method, --scripting-backend
- Cache key generation for Unity builds
- Support for Unity Hub multi-version installations

### Must NOT Have (Guardrails)
- NO automated license activation — workers must have pre-activated Unity
- NO Unity Hub installation/management — assume Editor already installed
- NO Docker-based Unity — license complexity makes this impractical for v0.5.0
- NO asset import distribution — whole-project builds only
- NO Addressables/AssetBundle granular builds — defer to v0.6.0
- NO breaking changes to C++ or Flutter builds — preserve existing functionality
